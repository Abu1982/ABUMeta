# 首发操作步骤

版本：`v1.0.04`

本文档用于指导 `D:\ABUMeta` 在 GitHub 上完成首次公开发布。

## 1. 首发目标

本次首发的目标不是发布主开发仓的全部能力，而是发布一个：

- 可运行
- 可测试
- 可展示
- 可继续二次开发
- 已做基本开源审计与版本治理的演示型公开仓库

## 2. 推送前确认

在真正创建 GitHub 仓库前，先确认以下文件已经齐备：

- `README.md`
- `LICENSE`
- `THIRD_PARTY_NOTICES.md`
- `CONTRIBUTING.md`
- `SECURITY.md`
- `更新日志.MD`
- `GITHUB_REPOSITORY_METADATA.MD`
- `公开前最终复核清单.MD`
- `RELEASE_NOTES_v1.0.03.MD`

并确认当前 Git 标签链完整：

- `v1.0.0`
- `v1.0.01`
- `v1.0.02`
- `v1.0.03`
- `v1.0.04`

## 3. 创建 GitHub 仓库

### 3.1 GitHub 页面创建方式

1. 登录 GitHub
2. 点击右上角 `New repository`
3. 仓库名填写：`ABUMeta`
4. 仓库可见性按你的需要选择：
   - 公开：`Public`
   - 若还想最后内部确认一次，可先 `Private` 再改 `Public`
5. 不要勾选：
   - `Add a README file`
   - `Add .gitignore`
   - `Choose a license`

原因：

- 本地仓库已经有完整的 README、LICENSE 和版本历史
- 避免远端初始化文件和本地历史冲突

### 3.2 推荐仓库设置

- Repository name：`ABUMeta`
- Description：使用 `GITHUB_REPOSITORY_METADATA.MD` 中的“仓库简介文案（短）”
- Website：可先留空
- Visibility：`Public`

## 5. 首次推送顺序

### 5.1 添加远端

将下面命令中的 `<your-username>` 换成你的 GitHub 用户名或组织名：

```powershell
git remote add origin https://github.com/<your-username>/ABUMeta.git
```

如果你使用 SSH：

```powershell
git remote add origin git@github.com:<your-username>/ABUMeta.git
```

### 5.2 首次推送主分支

```powershell
git push -u origin main
```

### 5.3 推送全部版本标签

```powershell
git push origin --tags
```

## 6. GitHub 仓库设置建议

推送完成后，建议在仓库 Settings 中完成以下设置：

### 6.1 General

- 启用 Issues
- 启用 Projects：按需
- 启用 Discussions：如果你希望公开讨论
- 保留 Wiki：可选

### 6.2 Pull Requests

- 启用 squash merge：建议开启
- 启用 auto delete head branches：建议开启

### 6.3 Branches

为 `main` 添加基础保护（如果仓库开始协作）：

- Require a pull request before merging
- Require status checks to pass before merging（后续再接 CI 时开启）

### 6.4 Security

- 启用 Dependabot alerts：建议开启
- 启用 secret scanning：若可用，建议开启

## 7. Release 发布顺序

建议首发时不要直接从 `v1.0.0` 发 Release，而是按“当前最完整首发状态”发布：

- 当前推荐首发版本：`v1.0.04`

原因：

- `v1.0.0`：开源演示版基线
- `v1.0.01`：依赖拆分与 GitHub 社区文件补齐
- `v1.0.02`：GitHub 元数据与首发文案收口
- `v1.0.03`：最终公开前复核完成
- `v1.0.04`：首发操作清单补齐，适合作为真正对外首发锚点

## 8. 首发 Release 创建步骤

### 8.1 在 GitHub 页面操作

1. 进入仓库首页
2. 点击 `Releases`
3. 点击 `Draft a new release`
4. 选择标签：`v1.0.04`
5. Release title 直接使用：

```text
v1.0.04 - GitHub 首发操作清单与发布就绪版
```

6. Release notes 正文建议使用本地最终稿，或在下一节的正文基础上粘贴

### 8.2 推荐 Release 正文

```md
## 本次更新

- 补齐 GitHub 首发操作清单，明确仓库创建、About、Topics、首发推送和 Release 发布顺序
- 完成开源演示版的版本治理、许可证说明、第三方审计、社区文件和最终公开前复核
- 当前仓库已具备最小可演示运行链、最小 smoke 测试与可回滚版本标签链

## 当前仓库状态

- 已具备最小可运行演示能力
- 已具备 smoke 测试
- 已具备 `LICENSE`、`README.md`、`THIRD_PARTY_NOTICES.md`、`CONTRIBUTING.md`、`SECURITY.md`
- 已具备 Issue 模板和 Pull Request 模板
- 已完成对公开内容的最终复核

## 验证结果

- 使用副本虚拟环境运行 `tests/test_smoke_demo.py`
- 测试结果：`4 passed`
- 当前发布标签：`v1.0.04`

## 注意事项

- 本仓库是开源演示版，不包含主开发仓数据库、抓取结果、缓存和历史运行资产
- 浏览器抓取、向量检索和外部 LLM 服务属于可选增强能力
- 详细第三方说明请见 `THIRD_PARTY_NOTICES.md`
```

## 9. 推送后的公开检查

发布完成后，建议按顺序检查：

1. 仓库首页是否正常显示 `README.md`
2. About 区域 description 是否正确
3. Topics 是否完整
4. `LICENSE` 是否被 GitHub 正常识别
5. Releases 页面是否能看到 `v1.0.04`
6. Tags 页面是否能看到完整历史：
   - `v1.0.0`
   - `v1.0.01`
   - `v1.0.02`
   - `v1.0.03`
   - `v1.0.04`
7. Issue 模板和 PR 模板是否生效
8. README 中命令示例是否显示正常

## 10. 首发后建议做的事

### 10.1 立即可做

- 设置仓库头像 / Social Preview
- 检查 Dependabot 和 Security 设置
- 置顶 `README.md` 中最重要的安装和运行入口

### 10.2 一周内建议完成

- 补充 FAQ
- 补充一张架构图
- 增加一个截图或 GIF 演示
- 若开始接收 PR，可增加 `CODE_OF_CONDUCT.md`

## 11. 最终执行顺序（极简版）

如果你只想按最短路径完成首发，直接按下面顺序做：

1. GitHub 创建空仓库 `ABUMeta`
2. 填写 About 区域 description 和 topics
3. 本地执行：

```powershell
git remote add origin https://github.com/<your-username>/ABUMeta.git
git push -u origin main
git push origin --tags
```

4. GitHub 新建 Release，选择标签 `v1.0.04`
5. 粘贴本地准备好的 Release 正文
6. 检查首页 README、LICENSE、Tags、Issue 模板、PR 模板
